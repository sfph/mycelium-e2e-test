# CFN `/decide` tracing: what each repo's instrumentation gave us

| Repo | What it gave us | Did it pay off? |
|---|---|---|
| `ioc-cognition-fabric-node-svc` | CFN-side timing envelope on `/decide`: per-request middleware + FastAPI dependency-resolution timing, a wall-clock header read so we could measure HTTP wire delay from Mycelium to CFN's middleware, and an event-loop lag sampler that captures stack snapshots when the loop wedges. | ✅ The wire-delay header was the first signal pointing the latency *outside* FastAPI. The loop-lag sampler with stack snapshots named the residual wedgers (LiteLLM async success hook, concept-graph neighbor lookups). |
| `ioc-cfn-cognition-engines` | Thread-dispatch breakdown around the engines' `asyncio.to_thread` calls: time spent waiting for a worker thread to start, time doing the actual work, and time resuming on the event loop after the thread finished. | ✅ **The smoking gun.** Wait-for-thread p95 = 75s while in-thread work was ~1ms told us instantly this was thread-pool exhaustion, not slow code. Without it, we'd have spent weeks optimizing the wrong thing. |
| `mycelium-io/mycelium` | Mycelium-side view of the `/decide` call: HTTP call breakdown (connect, request, response, JSON parse, close), Mycelium-side event-loop lag sampler, and capture of CFN's wire/loop-lag headers into the per-round trace. | ✅ Bracketed CFN cleanly — let us prove the latency was server-side, not client-side network or parsing, and gave us a Mycelium-side loop-lag baseline to cross-correlate against CFN's. |
| `sfph/mycelium-e2e-test` | Analyzer that aggregates per-round trace JSON into p50/p95/max distributions across a batch and surfaces a long-rounds detail block. | ✅ How we turned hundreds of per-round JSON blobs into the population-level pattern that revealed the bottleneck and later validated the fix at scale. |
